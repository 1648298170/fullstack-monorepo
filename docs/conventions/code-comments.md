# 代码注释规范

本仓库要求新增代码提供必要的中文注释，让维护者能够快速理解模块用途、关键流程和扩展
边界，避免核心逻辑成为只能靠逐行调试理解的黑盒。

## 必须注释的内容

- 文件或模块承担的主要职责。
- 不容易从函数名直接判断的关键流程。
- 目录、状态、构建或事务处理中的设计原因。
- 框架限制、业务边界和后续扩展点。
- 兼容性处理、特殊输入校验和容易误改的代码。
- 测试夹具、测试隔离方式以及测试重点。

示例：

```ts
// Page 只负责页面骨架；路由、权限和布局包含业务语义，因此保持手动注册。
function planPage() {
  // ...
}
```

## 不推荐的注释

不要逐行翻译代码，也不要添加没有信息量的注释：

```ts
// 将 count 加一。
count += 1;
```

如果代码本身已经足够清晰，优先通过准确命名表达含义。注释应解释“为什么这样做”、
“这段代码保护什么”或“修改时需要注意什么”。

## 文件类型约定

- TypeScript、JavaScript、Vue、CSS、SCSS 和构建配置按需添加说明性注释。
- 测试代码需要说明测试隔离方式或关键行为，不要求为每个断言重复解释。
- JSON 文件不添加注释，避免破坏标准 JSON 解析。
- Markdown 文档通过正文直接说明，不需要为了满足本规范额外添加 HTML 注释。

## 新增代码要求

后续新增或修改代码时，提交前需要检查：

1. 新模块是否有职责说明。
2. 复杂函数是否说明关键流程和约束。
3. 特殊分支是否解释存在原因。
4. 新增测试是否能让维护者看懂它保护的行为。
5. 注释是否仍与代码一致，过期注释必须同步更新或删除。

注释不是数量指标。目标是让接手代码的人能快速建立正确的心智模型，同时避免用大量
无意义注释淹没真正重要的信息。
